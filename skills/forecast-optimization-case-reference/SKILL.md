# 目的

作为 `forecast-optimization-advisor` 的补充，把真实预测实验案例转成可迁移的建议模式。输出必须仍由当前实验的指标、切片、badcase、日志、代码和可执行接入点驱动；案例只能提供参考方向，不能替代证据。

# 输入

可接受输入：
- `optimization_suggestions.md`
- `report_context.json`
- `experiment_plan.yaml`
- 整体指标、场景指标、badcase 明细
- 代码结构分析、特征函数、CLI 参数和 `feature_cols` 路径
- 日志摘要、异常摘要、产物缺失或 ambiguous 说明
- 用户明确指定的优化手册或案例参考请求

仅在以下情况追问：
- 当前没有任何指标、切片、badcase、日志或代码证据
- 用户要求直接复制案例改动，但当前实验源码、字段或预测口径不支持
- 用户要求自动修改代码、训练模型或接数据库

# 工作流

1. 先使用或读取 `forecast-optimization-advisor` 的证据化建议；没有证据化建议时，先按该 skill 的规则收集证据。
2. 判断是否应加载案例参考：
   - 用户明确要求参考手册、真实实验、LGBM/LightGBM 案例；
   - 或当前证据显示为低频、零膨胀、T+N 销量/需求预测，并且需要案例化实验方向。
3. 读取 `references/package-lgbm-optimization-case.md`，只提取与当前证据匹配的模式。
4. 将当前证据映射到候选方向：特征补充、口径检查、分层校准、后处理保护、诊断产物补齐。
5. 对每个候选方向确认源码接入点：已有参数、函数、字段、配置或可修改策略区块。没有接入点时，建议先定位接入链路，不要发明空开关。
6. 输出 2-4 条建议；完整分析可输出更多，但每条必须包含“证据、案例参考、动作、验证、风险”。
7. 输出后用 `skills/forecast-optimization-advisor/scripts/validate_suggestions.py` 校验建议文档；脚本失败时再读取脚本正文调试。

# 规则

- 每条建议必须引用当前实验证据；不能只引用案例。
- 不要把案例中的业务名、字段名、模型名、数据路径或日期窗口当成当前实验事实。
- 不要默认当前实验就是套餐、外卖、T+2 或 LightGBM；只有用户或证据支持时才使用这些词。
- 当案例字段不可用时，输出“等价字段/接入点待确认”，不要编造字段。
- 建议必须服务用户 ask 中的主指标；如果用户指定指标名，先沿用 `forecast-optimization-advisor` 对指标定义的确认结果。
- 不要建议修改 actual 数据、评估脚本、指标定义或数据切分。
- 不要建议基线对比；只能基于当前已给出的实验结果和候选结果提出下一轮验证。
- 后处理建议必须带 guard，且只能使用预测时可获得字段分组。
- 校准或切片分组不得使用事后误差标签，例如 underestimate、overestimate。
- 如果 high-heat 或高目标切片已经高估，不要建议全局上调。
- 如果证据只显示产物缺失或路径模糊，建议只能围绕补齐证据和确认产物。

# 输出格式

```markdown
# 案例参考优化建议

## 建议 1：<方向>

- 证据：<当前实验中的指标/切片/badcase/日志/代码证据>
- 案例参考：<案例中相似模式，不作为当前结论>
- 动作：<可执行的下一步实验或排查动作>
- 验证：<如何用当前主指标、切片指标和 badcase 验证>
- 风险：<泄漏、不可用字段、接入点、过补偿或切片恶化风险>
```

# 质量检查

最终输出前确认：
- 当前实验证据不是空的。
- 每条建议都有“证据、动作、验证”。
- “案例参考”没有被写成当前实验事实。
- 没有复制案例中的固定业务路径、固定日期窗口或固定配置。
- 没有自动训练、自动改代码或上线判断。
- 没有基线对比建议。
