# 评测附录报告模板

```markdown
# 预测实验评测报告

## 1. 一句话结论

说明本次是 Full 模式还是 Diagnostic 模式。

## 2. 任务理解

说明用户想评测什么，以及本次关注日志、代码、指标、badcase 还是建议。

## 3. 实验目录扫描结果

列出识别到的代码、配置、日志、数据、prediction、actual、metrics。

## 4. 日志解析结果

说明日志是否可用、是否发现 error、warning、loss、metric、样本量和完成信号。

## 5. 代码结构分析

说明训练入口、评估入口、指标函数、特征模块、模型模块和输出线索。

## 6. 评测数据可用性

说明 prediction、actual 是否可用，是否存在 ambiguous。

## 7. 整体指标

仅 Full 模式输出。

## 8. 细粒度效果拆解

仅 Full 模式且场景指标可用时输出。

## 9. badcase 定位

仅 Full 模式且 badcase 可用时输出。

## 10. 优化建议

每条建议必须引用证据。有 scene_metrics 或 badcase 证据时，必须写成具体可执行的特征实验：

- 特征名或参数实验：例如扩展已有 `rolling_windows` 到 14/30、构建源码可接入的 `recent_trend_ratio`、验证已有 lifecycle 字段的分桶交叉。
- 构造口径：说明按哪些维度分组、使用哪些时间窗口或字段、如何分桶或交叉。
- 验证指标：说明复跑后看哪个用户目标指标、WAPE、Bias 或具体场景 Bias。
- 源码落点：说明改的是 trial 副本中的哪个参数、函数或 feature_cols 路径；不要声称未被审计通过的空开关已经生效。

禁止只写“围绕高误差场景优化”“针对 badcase 构造特征”等泛化建议。

## 11. 下一步动作

说明需要补充哪些文件、是否建议人工复核、是否需要继续实验。
```

## final_report.md 规则

`final_report.md` 不使用以上完整评测结构。最终报告必须使用 `core/reports.py` 的实验验证结论模板：

- rollback / 训练失败 / 评测失败：输出“本轮新实验方案无效，当前最佳方案仍然是 baseline”，再说明失败原因、baseline 可优化方向和下一轮建议。
- keep / 指标改善：输出“本轮新实验方案有效，当前最佳方案切换为修改后方案”，再说明本轮改动、改善指标、剩余问题和下一步建议。

最终报告禁止展开完整扫描结果、候选文件列表、逐条 badcase、完整 token 统计和过程性章节；这些内容只作为附录路径引用。
